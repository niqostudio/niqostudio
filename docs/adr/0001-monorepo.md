# ADR 0001: システムを1つの monorepo に置く

- ステータス: 採用
- 決定日: 2026-06-03

## 背景
NIQO STUDIO のシステムは複数のモジュールから成り、**モジュール同士は疎結合に設計している**（責務分離・固有規約は各 `CLAUDE.md`・跨ぎの直読依存を作らない）。リポジトリ構成として monorepo か multi-repo かを選ぶ必要があった。

一般に multi-repo が優位とされる点（CI / ビルドコンテキストの肥大化、モジュール間のライフサイクル差）は理解した上で、次を重視した：

- `niqostudio` GitHub org には**プロダクト成果物**も置く。システムを multi-repo で散らすと、**「システム用リポ」と「プロダクト成果物リポ」の境界が曖昧**になる。
- 開発・AI のコンテキスト上、システム全体を**1つの文脈で横断的に読み書き**できる利点が大きい。
- 屋号自身のシステムを**1つの単位として表現**したい。

## 決定
システムを構成する全モジュールを**1つの monorepo** に置く（現在のモジュール構成は [アーキテクチャ](../architecture.md) を参照）。**プロダクト（製品）は別リポ/private** に置き、org 内で「システム＝この monorepo」と明確に分ける。

multi-repo 優位とされる点は**仕組みで防御**する（疎結合設計は崩さない）：

- **CI/コンテキストの肥大化** → `.github/workflows/` を **paths でモジュール別に出し分け**、ジョブは対象モジュールだけを対象にする。
- **ライフサイクル差** → モジュール境界と paths トリガで独立反映（本番反映はモジュール別 Environment の承認ゲート）。
- **疎結合の維持** → モジュールは互いの実装でなく**契約**（core の schema / RLS・生成型）にだけ依存する。共有は最小限（公開定数＝root の `config.<env>.json`・規約＝`.claude/rules`・docs）で、コピー配布や跨ぎの直読依存は作らない。型は workspace（`@niqostudio/db-types`）で共有する。

## 影響
- org 配下で「システム＝この monorepo / プロダクト＝別リポ」の境界が明確になる。
- モジュールは疎結合を保ち、固有規約・固有 `.env` を持つため、将来の切り出しにも追従できる。
- 共有の正本は1か所に集約し、コピー配布や跨ぎの直読依存を持たない。
